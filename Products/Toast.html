Toast

Please send feedback and issues to support@mutantcows.com.

Description

Create a party of characters and setup tactics for each to fight against the AI for items and players for XP. Player has no input and fight relies on tactics defined by player before the fight.

There will be two modes of fighting, arena fighting against the AI and pvp fighting against other players. Arena fighting will be used to get new items and equipment for your party to use. No experience will be earned in these fights, and the level and quality of items and gear rewarded increase with the number of waves completed. To level your party and get be able to equip better gear, you will earn experience through pvp fights with other players.

The goal is to create a game that allows me to use the tactics and strategies similar to those found in Final Fantasy XII and Dragon Age to put your skills in strategy against others. Dream goals would be to create a match making and fight replay system similar to that found in Clash of Clans.

12/12/2014

Before I go much further developing Toast, I decided it was time to take time to clean up and refactor the game code before adding any additional complexity, such as the tactics engine. What this allows me to do is clean up the code by removing unused methods, variables, properties and such that get created and abandoned while testing ways of doing things.

Refactoring is simply the organization of code to make it easier to read, understand, and maintain. By providing relevant names to variables and methods that describe what it is used for or what it does, the ability to read and follow what the code is doing is dramatically increased, especially for those who didn't write the code. Taking methods with multiple chunks of logic or behavior and putting those chunks into well named methods of their own is another important refactoring technique. The third most common refactoring technique I employee is creating service classes from the code. A service class is created and used to handle item or tasks specific actions, such as getting all, checking health, creating a new item, getting the total count of created items, and other such actions that would be used anywhere in the game. The service provides a way to centralize the code instead of potentially having multiple ways of doing the same thing in different parts of the game code. Even if the methods in the service are only used in say one class file, it is still easier to find all of the methods for the item, such as a monster, instead of having to find this code and logic buried in methods. The final thing I like to do is expose certain variables to the Unity UI. This is incredibly useful for tweaking values when adjusting game play and balance, and can be done while the game is running in Unity instead of requiring the developer to update the variable value in code and relaunch the game to see its effect. 12/2/2014

For this phase I am working on what I consider to be the most important part of this game, the tactics system. The tactics system is what will differentiate your team from everyone else's and defines the actions your team does in a fight. The tactics system is basically a rules engine that is used to determine what actions occur and to whom based on certain criteria.

Two of the best examples of this is in Final Fantasy 12 and in Dragon Age: Origins. In Final Fantasy 12, you have a team of three. You control the movements and combat choices for one of the characters while the other two are on auto pilot using the rules you define for each one. In Dragon Age: Origins, you have a team of up to four characters and you can define the rules for all of the characters. You control the actions of only one of the characters at a time, but you can switch between any of them most of the time while the others use the rules you define for their formation, movements, and actions.

It took me a while to wrap my head around how I wanted to handle this, and especially in a way that would allow the ability to handle a large combination of rules without a lot of code repetition and would make it easy to add additional actions and conditions, or at least some sort of standard way of adding them. I finally realized that I needed to step back and really define what it was I was trying to do.

The notes above, yes I know you probably can't read my writing, helped me solidify the logic for the rule engine itself. Next I tried to figure out how I could work this programmatically. I possibly over engineered this a little bit, but I just can't think of another way to handle multiple items in a similar and predictable fashion.

Warning: I am going to discuss some more complicated programing concepts called design patterns to describe how I intend to implement the rules engine. So my thought is to include the name of a class for each of the values used as the target, condition, and action. This class name would be used with a factory using the class name as the flag. I would then use reflection to create an instance of that class, such as "AnyTeamMember" as the class for a target, and return that class. Each of these classes would conform to one of three interfaces that is used to define the three types of classes, target, condition, and action. Using the interface, I will use a strategy method to call the methods on the class returned. This would allow me to hide the implementation details for each class, provide a standard set of method that have to be included, and allow me to predictably handle each class. This opens it up to allow me to create any number of classes for each type that each do their own thing, and also provides a way for the player to build any combination of strategies without adding and additional line of code. So by using Interfaces, the Factory Pattern, the Strategy Pattern, and reflection I can follow the logic from the third page to handle each rule in turn.

I would love input on this, I am very curious to see how other people have done this or if I am blatently overlooking anything.
